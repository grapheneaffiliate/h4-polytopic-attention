# TR87
**Status:** 4/6 local, 1/6 API | **Tags:** keyboard | **Baselines:** 37, 30, 39, 29, 63, 119

## Mechanic
Rule rotation puzzle. Player sees a top row (antecedents) and bottom row (consequents). Rules define input→output sprite mappings. Player rotates rules (ACTION1/2) and moves cursor (ACTION3/4) to make the bottom row match what the rules predict from the top row.

## Win Condition
`bsqsshqpox()` returns True when all antecedent sprites can be matched to consequent sprites using the transformation rules in `cifzvbcuwqe`. Triggers animation then `next_level()`.

## Actions
- ACTION1: rotate selected rule counterclockwise (cycles through 7 sprite variants)
- ACTION2: rotate selected rule clockwise
- ACTION3: move cursor left through rules
- ACTION4: move cursor right through rules
- Step counter: `upmkivwyrxz` decrements each action. Lose when 0.

## Solver Logic
1. Extract rules from game state: `cifzvbcuwqe` = list of (input_sprites, output_sprites) pairs
2. Read top row (antecedents `zvojhrjxxm`) and bottom row (consequents `ztgmtnnufb`)
3. Apply rules to top row → compute expected bottom row
4. For each position in bottom row, compute how many ACTION1/2 cycles needed to rotate current sprite to target
5. Compute ACTION3/4 cursor moves between positions
6. Output minimal action sequence

## Game State Layout
- `on_set_level` (line 910): sorts all `nxkictbbvzt` sprites by (y, x)
- Sprites below background y-midpoint split into `zvojhrjxxm` (top row) and `ztgmtnnufb` (bottom row)
- Rules `cifzvbcuwqe` built from `iqrduxrukrk` divider sprites — sprite left of divider = input, right = output
- Random rotation applied to all sprites with seed `[4, 2, 1, 2, 12, 20]` (cosmetic only)
- Bottom row variants randomized with seed `[7, 7, 32, 18, 23, 11]` — this is what the player has to fix

## alter_rules Mode (Levels 4-5)
When `alter_rules=True`, the RULE sprites themselves (not just bottom row) get randomized:
```python
for zvojhrjxxm, ztgmtnnufb in self.cifzvbcuwqe:  # for each rule pair
    for zdwrfusvmx in (zvojhrjxxm, ztgmtnnufb):  # both input and output
        dkmtlqnzhy = fcnjuctdrc.randint(0, 6)     # random offset
        for i, ekifeojqhe in enumerate(zdwrfusvmx):
            for garumvurkj in range(dkmtlqnzhy):
                ekifeojqhe = self.wpbnovjwkv(ekifeojqhe, 1)  # cycle forward
            zdwrfusvmx[i] = ekifeojqhe
```
**Fix needed:** The solver already reads rules from `game.cifzvbcuwqe` which IS the post-randomization state. The bug is that the rule matching fails because `alter_rules` mode also changes how the cursor and cycling work (line 998-1010): the cursor iterates over ALL sprites in ALL rules (flattened), not just the bottom row. ACTION1/2 cycles the selected rule sprite, not the bottom row sprite.

## double_translation Mode (Level 3)
Rules come in two layers (A->B and B->C). The win check `bsqsshqpox` (line 1080):
1. Match top row segment against rule input
2. Get rule output (intermediate)  
3. Match intermediate against ANOTHER rule's input
4. Get second output = expected bottom row

Solver fix (implemented): apply rules twice by looking up each intermediate output in the rule table.
