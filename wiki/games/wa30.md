# WA30
**Status:** 0/9 unsolved | **Tags:** keyboard | **Baselines:** 125, 58, 259, 113, 499, 58, 186, 134, 132

## Mechanic
Sokoban-style crate pushing. Player moves on 64×64 grid, pushes crates to target positions. Step budget per level.

## Win Condition
`ymzfopzgbq()`: ALL target sprites have matching crates AND all crates not in exclusion set `zmqreragji`.

## Why Unsolved
- Baseline 125+ for level 0 — way too deep for any BFS variant
- Reset+replay: 80 nodes/s, reached depth 20 in 300s but need 125+
- Deepcopy BFS: 77 states/s, 3157 unique states in 120s
- State space IS small (3157 states with 5 actions) but solution path is very long

## Key Sprites
- `wbmdvjhthc`: player (at 32,48 level 0)
- `geezpjgiyd`: crates (3 crates in level 0)
- `fsjjayjoeg`: obstacles/walls
- Movement stride: 4 pixels per action

## What's Needed
An optimal Sokoban solver (IDA* with pattern database heuristic). The state is (player_pos, crate_positions) — compact enough for A* but needs a good heuristic. Manhattan distance to targets is the standard approach.

Alternatively: the state space is only 3157 — an abstract solver that models (player_grid_pos, crate_grid_positions) as tuples could BFS the entire space quickly. The issue is only the deepcopy/replay speed, not the state space size.
